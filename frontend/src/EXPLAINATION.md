# โครงสร้างโฟลเดอร์ใน Frontend

เพื่อให้ระบบขยายขนาดได้ (Scalable) และง่ายต่อการบำรุงรักษา เราได้แบ่งโครงสร้างตามหน้าที่ของแต่ละส่วน ดังนี้:

## 1. `pages/` (เพจหลัก)
**หน้าที่:** เป็นจุดเริ่มต้นของแต่ละเส้นทาง (Routes) ในแอปพลิเคชัน
- ไฟล์ในนี้จะถูกเรียกใช้ใน `App.js` เพื่อกำหนด URL (เช่น `/login`, `/admin`)
- **ความสัมพันธ์:** มักจะเรียกใช้ `layouts` หรือประกอบร่าง `features` และ `components` เข้าด้วยกันเพื่อสร้างหน้าเต็มๆ

## 2. `layouts/` (โครงสร้างหน้าตา)
**หน้าที่:** เป็น "Template" หรือโครงร่างหลักของกลุ่มหน้าเพจที่มีหน้าตาคล้ายกัน
- เช่น `AdminLayout` จะมี Sidebar และ Navbar สำหรับ Admin โดยเฉพาะ, `ContestLayout` จะมีแถบเมนูสำหรับการแข่งขัน
- **ความสัมพันธ์:** จะมี `<Outlet />` เพื่อให้ `pages` ต่างๆ มาแสดงผลภายในโครงสร้างนี้

## 3. `features/` (ความสามารถเฉพาะส่วน)
**หน้าที่:** เก็บส่วนประกอบ (Components) และ Logic ที่ผูกติดกับ "ฟังก์ชันธุรกิจ" เฉพาะอย่าง
- เช่น `features/admin` จะเก็บ Modal การเพิ่ม User, การแก้ไขโจทย์ ซึ่งเป็นฟังก์ชันที่ใช้เฉพาะในส่วนของ Admin เท่านั้น
- **การใช้งาน:** เพื่อไม่ให้โฟลเดอร์ `components` รวมทุกอย่างจนปนกัน เราแยกสิ่งที่ใช้เฉพาะเรื่องมาไว้ในนี้

## 4. `components/` (ส่วนประกอบที่ใช้ร่วมกัน)
**หน้าที่:** เก็บ UI Components ขนาดเล็กถึงกลางที่ **ไม่ผูกติดกับ Logic ธุรกิจ** และสามารถนำไปใช้ซ้ำได้ทั่วทั้งโปรเจกต์
- เช่น ปุ่มกด (Button), กล่องข้อความ (Modal), แถบเมนูด้านบน (Navbar) หรือตัวเลือกธีม (ThemeToggleButton)
- **การใช้งาน:** ถ้า UI นั้นสามารถใช้ได้ทั้งในหน้า Home, Admin และ Contest ควรจะเก็บไว้ที่นี่

## 5. `context/` (การจัดการสถานะส่วนกลาง)
**หน้าที่:** เก็บสถานะ (State) ที่ต้องใช้งานร่วมกันทั้งแอปพลิเคชัน เพื่อหลีกเลี่ยงการส่ง Props หลายชั้น (Prop Drilling)
- เช่น `AuthContext` เก็บข้อมูลผู้ใช้ที่ล็อกอิน, `ThemeContext` เก็บสถานะ Dark/Light Mode
- **ความสัมพันธ์:** ทุกส่วนของแอปสามารถเข้าถึงข้อมูลในนี้ได้ผ่าน `useContext`

## 6. `hooks/` (Logic ส่วนตัวที่ใช้ซ้ำได้)
**หน้าที่:** เก็บ Custom Hooks ของ React เพื่อแยก Logic ที่มีความซับซ้อนออกมาจากตัว UI
- เช่น `useProblemDetail` จัดการเรื่องการดึงข้อมูลโจทย์และสถานะโหลด, `useContestGuard` ตรวจสอบสิทธิ์การเข้าถึงการแข่งขัน
- **การใช้งาน:** ช่วยให้โค้ดใน `pages` หรือ `features` สั้นลงและอ่านง่ายขึ้น

## 7. `services/` (ตัวติดต่อ API)
**หน้าที่:** เป็นเลเยอร์สำหรับสื่อสารกับ Backend (API) โดยเฉพาะ
- เช่น `authService.js`, `problemService.js` จะรวมคำสั่ง `api.get`, `api.post` ไว้ในที่เดียว
- **ประโยชน์:** หากมีการเปลี่ยน URL ของ API หรือโครงสร้างข้อมูล เราสามารถแก้ไขได้ที่โฟลเดอร์นี้ที่เดียว

## 8. `utils/` (เครื่องมือช่วยเสริม)
**หน้าที่:** เก็บฟังก์ชันช่วยเหลือ (Helper Functions) และค่าคงที่ (Constants) ที่ไม่มีสถานะของ React
- เช่น `formatters.js` สำหรับแปลงวันที่หรือจัดการชื่อ Class สี, `constants.js` เก็บค่าคงที่ต่างๆ ของระบบ
- **การใช้งาน:** เป็นฟังก์ชัน Pure Logic ที่เรียกใช้ง่ายและนำไปใช้ซ้ำได้ทุกที่

---

### สรุปเปรียบเทียบ
| โฟลเดอร์ | หน้าที่หลัก | ระดับความเฉพาะเจาะจง |
| :--- | :--- | :--- |
| `pages` | ตัวเชื่อมต่อกับ Route (เชื่อม URL) | สูงมาก (1 ไฟล์ : 1 URL) |
| `layouts` | โครงสร้างพื้นฐาน (Shell) ของหน้า | ปานกลาง (แบ่งตามกลุ่มผู้ใช้) |
| `features` | ฟังก์ชันการทำงานเฉพาะเรื่อง | ปานกลาง (แบ่งตามโมดูล) |
| `components` | UI สำเร็จรูปสำหรับใช้ซ้ำ | ต่ำ (ใช้ได้ทุกที่) |
| `context` | ข้อมูลส่วนกลาง (เช่น User, Theme) | Global (ใช้ได้ทั้งแอป) |
| `hooks` | Logic การทำงาน (Stateful logic) | กลาง (ใช้ซ้ำตามจุดที่ต้องการ) |
| `services` | การส่งข้อมูลกับ Server (API) | เฉพาะทาง (Domain-based) |
| `utils` | ฟังก์ชันช่วยคำนวณ/จัดการข้อมูล | ต่ำ (ฟังก์ชันทั่วไป) |
